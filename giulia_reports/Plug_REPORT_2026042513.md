### Cycles

1. **Plug.Adapters.Test.Conn ↔ Plug.Conn ↔ Plug.Conn.Adapter ↔ Plug.Conn.Cookies ↔ Plug.Exception ↔ Plug.MIME** — 6-module cycle. Driven by the Plug.Conn ↔ Plug.Exception ↔ Plug.Conn loop (exceptions reference Conn for the `:status` field on the protocol; Conn references exceptions for sending error responses). Idiomatic for an HTTP plug library.
2. **Plug.Upload ↔ Plug.UploadError** — module ↔ error pair. Compile-time mutual reference (UploadError raised inside Upload). Idiomatic.
3. **Plug.Router.InvalidSpecError ↔ Plug.Router.Utils** — module ↔ error pair. Same shape.
4. **Plug.Session ↔ Plug.Session.Store** — Session module ↔ behaviour module. Session calls into Store; Store typespec references Session.t. Idiomatic.

**Severity: P3 (informational).** None of these are architectural failures — they're the standard error-module + protocol-with-typespec patterns. The defprotocol/exception cycles are unavoidable when a `defprotocol` for `:status` references the type that owns the protocol's data.
